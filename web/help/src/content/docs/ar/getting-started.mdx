Safeparts يحوّل سراً حساساً واحداً إلى عدة **حصص استرداد**. يمكنك استعادة السر لاحقاً
باستخدام **k من n** حصص.

:::caution
الحصص حساسة. لا تضع حصصاً حقيقية في الدردشة أو التذاكر أو الـ issues أو السجلات.
عند التوثيق أو الاختبار استخدم أسراراً تجريبية.
:::

## ما المشكلة التي يحلها هذا؟

أغلب الناس ينسخون السر بوضع نسخة واحدة. هذا يخلق نقطة فشل واحدة:

```
السر -> نسخة احتياطية واحدة

إذا سُرقت النسخة: انكشف السر
إذا ضاعت النسخة: تعذر الاسترداد
```

مع Safeparts يمكنك توزيع المخاطر على أشخاص وأماكن متعددة:

```
السر -> n حصص

المهاجم يحتاج k حصص
أنت تحتاج k حصص
```

## النموذج الذهني k من n

- `n` = عدد الحصص التي تُنشئها
- `k` = عدد الحصص المطلوبة للاسترداد

مثال: `k=2, n=3`

```
تُنشئ 3 حصص.
أي حصتين يمكنهما الاسترداد.
يمكنك فقدان حصة واحدة وما زلت تستطيع الاسترداد.
```

### اختيار k و n (قاعدة عملية)

اختر `k` بناءً على **من يجب أن يوافق على الاسترداد**، ثم اختر `n` بناءً على **عدد الحصص التي يمكنك تحمل فقدانها**.

قاعدة بسيطة:

`n = k + (عدد الحصص التي يمكنك تحمل فقدانها)`

## أمثلة حسب الحالة

كل مثال يتضمن:

- قيم مقترحة لـ `k` و `n`
- طريقة توزيع (من/أين)
- الواجهة الموصى بها (واجهة الويب / سطر الأوامر / واجهة طرفية)

### 1) مفتاح استرداد مدير كلمات المرور (شخصي)

- المقترح: `k=2, n=3`
- التخزين: حصة مع شخص موثوق + حصة في خزنة/صندوق أمانات + حصة في مزود/مكان مختلف
- الواجهة: واجهة الويب

### 2) نسخة احتياطية لعبارة الاسترداد (فرد)

- المقترح: `k=3, n=5` للقيمة العالية؛ `k=2, n=3` للإعدادات الأبسط
- التخزين: مواقع فيزيائية مختلفة؛ ويفضل حصة offline (ورق/معدن)
- الواجهة: واجهة طرفية (جهاز offline) أو واجهة الويب

### 3) الإرث الرقمي / التخطيط للتركات (عائلة)

- المقترح: `k=2, n=4`
- التخزين: شريك/زوج + منفذ/محامٍ + موقع فيزيائي آمن + شخص إضافي اختياري
- الواجهة: واجهة الويب

### 4) حساب منزلي مشترك (عائلة)

- المقترح: `k=2, n=3`
- التخزين: شخصان بالغان يحتفظ كل منهما بحصة؛ الحصة الثالثة نسخة offline مختومة
- الواجهة: واجهة الويب

### 5) أكواد استرداد 2FA (شخصي)

- المقترح: `k=2, n=3`
- التخزين: تقسيم الأكواد بحيث لا يؤدي اختراق مدير كلمات مرور واحد إلى كشف كل شيء
- الواجهة: واجهة الويب

### 6) استرداد حساب مشرف النطاق/DNS (شركة صغيرة)

- المقترح: `k=2, n=3` أو `k=3, n=5`
- التخزين: توزيع على أدوار مختلفة (تشغيل + أمن) وحصة offline
- الواجهة: سطر الأوامر (عملية قابلة للتكرار) أو واجهة الويب (مرة واحدة)

### 7) رمز وصول GitHub/GitLab (مطور)

- المقترح: `k=2, n=3`
- التخزين: مدير كلمات مرور + مخزن أسرار CI + نسخة offline
- الواجهة: سطر الأوامر

### 8) اعتماد إنتاج للطوارئ (فريق)

- المقترح: `k=3, n=5`
- التخزين: أشخاص/أدوار مختلفة؛ وثّق خطة استرداد؛ قم بالتدوير بعد الاستخدام
- الواجهة: سطر الأوامر

### 9) حسابات خدمة مشتركة (فريق صغير)

- المقترح: `k=2, n=4`
- التخزين: مالك + تشغيل + أمن + نسخة offline
- الواجهة: سطر الأوامر للتدوير، وواجهة الويب للإعداد

### 10) تسليم عميل (وكالة)

- المقترح: `k=2, n=3`
- التخزين: خزنة الوكالة + خزنة العميل + ضمان اختياري
- الواجهة: واجهة الويب لاجتماع التسليم

### 11) استمرارية المؤسسين (شركة ناشئة)

- المقترح: `k=2, n=4`
- التخزين: مؤسسان + مستشار قانوني + نسخة offline
- الواجهة: سطر الأوامر لسهولة التكرار التشغيلي

### 12) امتثال: فصل المهام (مؤسسة)

- المقترح: `k=2, n=4` أو `k=3, n=5`
- التخزين: توزيع على أدوار مختلفة بحيث لا يستطيع شخص واحد الاسترداد بمفرده
- الواجهة: سطر الأوامر
