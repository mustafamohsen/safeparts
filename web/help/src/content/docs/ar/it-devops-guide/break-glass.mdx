يُستخدم سر break-glass نادراً وغالباً يمنح وصولاً مرتفع الصلاحية. يجب أن يكون الإجراء قصيراً بما يكفي للتنفيذ أثناء حادثة، ومنضبطاً بما يكفي لمنع الاستخدام العادي.

## نموذج حيازة مثال

يحتاج فريق منصة إلى وصول طارئ لحساب cloud إذا تعطلت أنظمة الهوية العادية. يختارون <span class="token" dir="ltr">k=3, n=5</span>.

حيازة مقترحة:

- الحصة 1: قائد الأمن.
- الحصة 2: قائد العمليات.
- الحصة 3: مدير الهندسة.
- الحصة 4: خزنة الشركة offline.
- الحصة 5: راعٍ تنفيذي أو أمين امتثال.

يلزم ثلاثة مشاركين مستقلين. غياب شخص واحد لا يمنع الاسترداد، وحامل واحد منكشف لا يستطيع الاسترداد وحده.

## إجراء الإعداد

<Steps>
1. أنشئ الاعتماد أو صدّره في بيئة متحكم بها.
2. قسّمه بالعتبة والترميز وسياسة عبارة المرور المختارة.
3. أعط كل أمين حصته فقط.
4. سجّل الحيازة وجهات التواصل والعتبة والترميز.
5. نفّذ تدريباً اصطناعياً بالعملية نفسها.
6. دمّر الملفات المؤقتة التي تحتوي الاعتماد أو كل الحصص معاً.
</Steps>

أمر مثال:

```bash
safeparts split \
  -k 3 \
  -n 5 \
  -e mnemo-words \
  --passphrase-file passphrase.txt \
  --in cloud-admin-recovery.txt \
  --out shares.txt
```

## إجراء الاسترداد

<Steps>
1. افتح سجل الحادثة أو التغيير الطارئ.
2. أكد الموافقة حسب السياسة.
3. اجمع على الأقل <var>k</var> حصص في بيئة الاسترداد.
4. ادمج الحصص واستعد الاعتماد.
5. استخدم الاعتماد للمهمة المعتمدة فقط.
6. دوّر الاعتماد المستعاد أو عطّله.
7. أنشئ تقسيماً جديداً في Safeparts للاعتماد البديل.
</Steps>

أمر مثال:

```bash
safeparts combine \
  --passphrase-file passphrase.txt \
  --in selected-shares.txt \
  --out recovered-cloud-admin.txt
```

:::danger
بعد استرداد break-glass، افترض أن الحصص المجموعة انكشفت. دوّر الاعتماد الأساسي وأصدر تقسيماً جديداً.
:::

## سجل التدقيق

سجّل:

- من وافق على الاسترداد؛
- أي أمناء شاركوا؛
- متى بدأ الاسترداد ومتى انتهى؛
- أي اعتماد أو نظام تم استرداده؛
- ما التدوير الذي تم بعد ذلك.

لا تسجّل نص الحصص أو عبارات المرور أو لقطات الشاشة للحصص أو الاعتمادات المستعادة.
