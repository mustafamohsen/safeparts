Use Safeparts in operations when recovery should require cooperation. It is best for escrow, emergency access, recovery drills, and controlled handoff. It is not a replacement for a secrets manager that injects routine service credentials.

:::caution
Operational records should show custody, approvals, timestamps, and rotation. They should not include real share text, passphrases, or recovered secrets.
:::

## Where it fits

Good fits:

- break-glass cloud or infrastructure credentials;
- password manager recovery keys;
- release signing recovery material;
- offline recovery tokens;
- small-team emergency access.

Poor fits:

- high-frequency runtime secrets;
- service-to-service credentials that need automatic online access;
- workflows where one CI system would hold enough shares to recover during every deployment.

## Minimum runbook questions

A safe runbook answers these before a real split happens:

1. What exact secret is being split?
2. What threshold is required?
3. Who or what holds each share?
4. Who approves recovery?
5. Where does recovery happen?
6. What gets rotated after recovery?

If recovery gathers enough shares to meet the threshold, treat the event as sensitive even when the process was authorized.
